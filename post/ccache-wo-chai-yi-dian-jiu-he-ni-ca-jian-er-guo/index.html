CCache 我差一点就和你擦肩而过

2019-04-19

可能有些同学还不知道什么是 CCache，简单说一些 CCache 是针对于 C/C++/OC 的一套优化缓存的编译系统

  • Xcode 编译缓存

    • archive 模式会忽略缓存

    • DerivedData 目录经常出问题

  • CCache

    • archive 不会忽略缓存
    • 命中率基本稳定

使用 CCache 大体上可以提升70%的编译时间。

我之前也看过网上介绍 CCache 加速编译时间的文章，但是也是一时间的冲动。最后发现用到的一些第三方库也用到了@import,也没在网上的文章中发现解决办法就放弃了。

其实加速编译有很多种方法，但是影响最小的应该是升级硬件。前段时间也申请了 Mac 打包机器，但是阴差阳错的申请失败。

但是可喜的是重新申请的买的都是新型的灰色 Mac 机器，喜闻乐见呀！。

我是今年把公司一个台式机安装成了黑苹果，从我电脑打包转移到黑苹果上面。显然硬件很差，打包起来十分吃力，苦等新电脑的下来。

既然暂时硬件升级指望不上，那就从软件入手。其中提升最大的就是接入 CCache 了，那就再次的尝试接入。

安装 CCache

第一步就是在电脑上面安装 CCache，因为我们需要在我们的电脑上面进行测试。

brew install ccache

网上很多教程都是这个命令进行安装，但是很多人连 HomeBrew都没有安装，导致命令会报错。后来我写了一个脚本放在 Xcode 新建的脚本命令里面。

install_ccache

#!/bin/bash
echo "开始安装 CCache"
if command -v ccache >/dev/null 2>&1; then
	echo "CCache 已经安装"
else
	if command -v brew >/dev/null 2>&1; then
		echo "HomeBrew已经安装"
		brew install ccache
	else
		echo "HomeBrew没有安装"
		ruby_path=`which ruby`
		echo $ruby_path
		$ruby_path -e "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/master/install)"
		brew install ccache
	fi
fi

后来小伙伴告诉我运行会报错，最后发现运行这个脚本安装 HomeBrew需要手动输入密码获取权限的。没办法，这个脚本只能从 Xcode 进行移除了。

难道还让其他的小伙伴按照网上的教程真的一步步的配置安装吗？那岂不是太麻烦了。因为 ccache-clang已经伴随着代码托管更新在小伙伴电脑上面，所以小伙伴就不用配置 CCache配置文件了。

既然没办法在 Xcode运行脚本，我们可以新建一个脚本，自动化运行，这叫脚本大法。

setup.sh

chmod 777 ccache-clang
chmod 777 ccache-clang++
chmod 777 ZHCCache
chmod 777 install_ccache
bash install_ccache

上面的 Shell脚本代码里面就 ZHCCache是我写的一个基于 Swift的一个 Command Line Tool程序，程序的源代码托管在下面的链接。

ZHCCache(修改@import 工具)

这样其他小伙伴只需要运行下面命令就可以成功接入 CCache

cd setup.sh目录
bash setup.sh

项目设置

电脑我们已经安装好了 CCache，接下来我们要做的就是项目设置 CCache

项目关闭 Enable Module

因为我这边自动打包需要就新建一个 AdHoc的配置，基本配置和 Debug一样但是证书是和 Release一致的。

新建 Macro CC

我一致很奇怪为什么可以走我们新建的 ccache-clang脚本，这个配置就是让 Xcode 自动运行路径脚本，所以我们才要赋予 777权限

项目已经配置好了，关闭了 Enable Module。我们要删除工程里面用到所有 @import的代码换成 #import <>

可能我们要把系统自动引入的库收到导进来。

因为我们项目里面包含的@import代码很少，很快就修改完毕。引入的库手动也很快的添加完毕。

修复第三方的@import

我们项目用到了很多的 Cocoapods 第三方，其中也少不了第三方用到了 @import。但是改动代码之后，下次 pod更新再次回滚了。

怎么能每次编译之前，把第三方里面的 @import替换掉。我第一想到是用脚本，shell语法了解甚少，决定还是用 Swift Command Line Tool进行修改。

写出来的工具就是 ZHCCache用来修改我们的第三方库的 @import.关键怎么让脚本知道我要修改那些，怎么才能容易扩展。

$PWD代表获取工程文件所在的目录

但是比如我们项目用到了很多的 Swift源码的 CocoaPods第三方库，会在编译的时候生成 -Swift.h类似的桥接文件。

有些桥接文件里面是存在 @import的，我们不能用上面指定目录设置修复。因为动态生成的，路径我们没办法获取。

就在我心灰意冷的时候我发现在 xcodebuild -showBuildSettings命令输出里面，包含了很多信息，还真的让我用曲线救国的思想找到了。

修改 CocoaPods 自动生成-Swift.h 桥接文件@import

$PODS_CONFIGURATION_BUILD_DIR/GlobalegrowIMSDK/GlobalegrowIMSDK.framework/Headers/GlobalegrowIMSDK-Swift.h
>>>
#if __has_feature(modules)
@import CoreGraphics;
@import Foundation;
@import ObjectiveC;
@import Photos;
@import UIKit;
#endif
>>>
#import <CoreGraphics/CoreGraphics.h>
#import <Foundation/Foundation.h>
#import <Photos/Photos.h>
#import <UIKit/UIKit.h>

PODS_CONFIGURATION_BUILD_DIR这个作为我们的关键词用于我们动态的获取所在的目录。

并且build 和 archive 的目录还是不一致的。

编译修复@import

我们需要在编译工程源代码之前运行我们的脚本修复。

脚本代码

# 修复文件存在@imprt问题
$PWD/ZHCCache PWD="$PWD" OTHER_CFLAGS="$OTHER_CFLAGS" FIX_PATH="$PWD/CCacheHeaderFix" CONFIGURATION=${CONFIGURATION}

ZHCCache 脚本我会放在 ccache-clang同级目录，CCacheHeaderFix是我们存在修复的配置文件目录。CONFIGURATION这个参数是获取当前的工程配置，因为我们在 Debug模式下面不修改，不然就会重复编译问题。

CocoaPods 有没有必要使用 CCache

对于我们的项目其实也用到了很多的 CocoaPods，我们也想在 Archive 使用缓存加快时间。我们可以使用下面的进行修改 podfile

  # 支持 CCache 的 Pod 数组
  @support_ccache_pod_names = []
  
  post_install do |installer|
      installer.pods_project.targets.each do |target|
          # 添加 POD 支持 CCache
          if @support_ccache_pod_names.include? target.name
              target.build_configurations.each do |config|
                  puts config.build_settings
                  if config.name != 'Debug'
                      config.build_settings['CC'] = '$(PODS_ROOT)/ccache-clang' # 替换为你的 ccache-clang 文件路径
                      config.build_settings['CLANG_ENABLE_MODULES'] = 'NO'
                  end
              end
          end
      end
  end

后来我试了，把 CocoaPods里面的 Enable Module关闭之后，把需要引用的系统库添加到工程里面，还是一样的报错。

只有把对应的 CocoaPods配置里面添加对应的系统库添加进去才不报错，和网上说的不一致，所以因为这个问题暂时没把 CocoaPods支持 CCache。

  • 其实可以通过动态修改下发的 PodSpec 文件(猜测可行)

  • 把第三方库托管成本地私有库 改掉本地的 PodSpec

切换分支编译优化

因为在同一个项目切换分支修改功能，Xcode 会频繁的编译。对于针对不同的分支修改 BUG和新增功能，就会变得十分的麻烦，需要频繁的编译等待。

对于这个我目前写了一个脚本，针对于不同的分支有不同的编译路径，这样就解决了频繁切换分支，进行频分编译的问题。

这样间接的在日常的开发工作中，提升了不少的编译时间，这也算是一种编译时间的优化吧。
